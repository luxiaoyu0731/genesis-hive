# Genesis Hive — Claude Code 启动全流程

## 第零步：环境准备

在终端中执行：

```bash
# 1. 确认 Claude Code 已安装
claude --version

# 2. 进入项目目录
cd ~/你的路径/GenesisHive

# 3. 确认文件结构就位
tree -a -L 2
# 应该看到：
# .claude/
#   ├── CLAUDE.md
#   ├── hooks/block-destructive.sh
#   └── settings.json
# SKILL.md
# 可行性验证报告.md
# Genesis_Hive_...项目规划.docx

# 4. 初始化 git（Claude Code 需要 git 仓库才能工作）
git init
git add -A
git commit -m "init: Genesis Hive project skeleton with SKILL.md and safety configs"

# 5. 准备好你的 API Keys（至少需要一个，推荐准备多个）
#    - OpenAI API Key（GPT-4o-mini / GPT-4o）
#    - Anthropic API Key（Claude Sonnet / Opus）
#    - 可选：Google Gemini / DeepSeek API Key
#    把它们放在 .env 文件里（后面 Claude Code 会帮你创建）
```

## 第一步：启动 Claude Code

```bash
# 在 GenesisHive 目录下启动
claude
```

启动后 Claude Code 会自动读取 `.claude/settings.json`（权限规则）和 `.claude/CLAUDE.md`（行为准则）。你不需要手动告诉它这些文件的存在。

---

## 第二步：让它读 SKILL.md 并搭骨架

**你的第一条 prompt（复制粘贴即可）：**

```
读一下 SKILL.md，然后按 Step 1 初始化项目。具体要求：

1. 创建完整的目录结构（backend/engines, backend/core, backend/tools, backend/api, frontend/src/components, frontend/src/pages, templates, cases）
2. 初始化 Python 后端：创建 pyproject.toml 或 requirements.txt，安装 fastapi, uvicorn, langgraph, langchain-core, openai, anthropic
3. 初始化前端：npm init + 安装 react, typescript, tailwindcss, d3, framer-motion
4. 创建 .env.example 模板（列出需要的 API Key 变量名，不要填真实值）
5. 实现 HiveState（TypedDict）定义，放在 backend/core/state.py
6. 实现 llm_service.py 的多模型适配层——统一接口，根据 model 参数路由到不同 Provider
7. 实现 message_bus.py 的消息总线骨架——按 SKILL.md 中的消息格式定义数据结构

搭完后给我看 tree 输出和各文件的代码。不要写业务逻辑（L1-L5引擎），只搭基础设施。
```

**验收标准**：看到完整目录树，state.py 中的 HiveState 包含 SKILL.md 里定义的所有字段（包括 agent_configs_hash、debate_history_full、evolution_cycle），llm_service.py 能根据模型名路由到 OpenAI/Anthropic/其他。

## 第四步：L2 Spawner（Agent 动态生成引擎）

```
现在实现 Step 3——L2 Spawner 引擎。

功能要求：
1. 接收 Decomposer 输出的 task_graph
2. 为每个子任务动态生成 Agent 配置（JSON Schema 约束输出）
3. 必须体现三层认知多样性：
   - 模型异构：调研型→GPT-4o-mini，分析型→Claude Sonnet，魔鬼代言人→Gemini/DeepSeek
   - 信息源隔离：不同 Agent 分配不同搜索策略
   - 思维框架差异：在 system_prompt 中嵌入 JTBD / Gartner Hype Cycle / Unit Economics / Pre-mortem
4. 总是生成一个魔鬼代言人 Agent
5. 工具分配逻辑：从 tool_pool 按能力需求自动匹配

接上一步的 Decomposer 输出做端到端测试，给我看生成的全部 Agent 配置。
重点检查：每个 Agent 的 model 字段是否真的不同，思维框架是否真的不同。
```

## 第五步：L3 Executor（并行执行引擎）

```
现在实现 Step 4——L3 Executor 引擎。

功能要求：
1. 用 asyncio.gather() 实现 IO 密集型并发
2. 尊重任务依赖关系（有依赖的 await 前置任务）
3. 支持增量执行模式（对比 agent_configs_hash 判断是否变更）
4. 每个 Agent 记录执行日志（actions, preliminary_result, confidence, tokens_used, time_ms）
5. 每个 Agent 有独立的 token 预算

先用 mock LLM 调用验证并发逻辑：
- 3 个无依赖任务应该几乎同时开始
- 有依赖的任务应该等待前置完成后才开始
- 打印每个任务的开始和结束时间来证明并发

Mock 验证通过后再接真实 LLM。
```

## 第六步：L4 Council（圆桌辩论引擎）——最核心

```
现在实现 Step 5——L4 Council 圆桌辩论引擎。这是整个项目最核心的部分，仔细看 SKILL.md。

功能要求：
1. Phase A 独立呈述：每个 Agent 展示初步结果
2. Phase B 圆桌辩论：消息总线交换观点，发言标注类型标签（质疑/支持/反驳/补充/提问）
3. Phase C 修正更新：Agent 基于辩论反馈修正结论
4. Phase D 共识检测：裁判 LLM 方案（NOT cosine similarity）
   - 结构化提取每个 Agent 的结论 JSON
   - 独立裁判模型判断核心立场是否一致
   - 输出 consensus_reached + consensus_type + recommendation
5. Phase E（新增的）辩论历史摘要压缩：
   - 用 GPT-4o-mini 将本轮完整辩论压缩为结构化摘要
   - 摘要存入 debate_history，原文存入 debate_history_full

写完后跑完整的 Decomposer→Spawner→Executor→Council 端到端测试。
重点检查：裁判 LLM 的输出是否能区分"同话题"和"同结论"。
```

## 第七步：L5 Evolver（动态重组引擎）

```
现在实现 Step 6——L5 Evolver 动态重组引擎。

功能要求：
1. 能力缺口触发：裁判 LLM 判断 capability_gap_detected 时激活
2. 低贡献检测（量化标准）：被引用次数 + 结论变更影响 + 信息增量，contribution_score < 0.2 且引用为 0 则退场
3. 进化次数上限：max_evolution_cycles = 2，超过后强制 synthesizer
4. 重组日志：记录每次变化的原因和决策依据
5. 进化后回到 Spawner 时，Executor 用增量执行（只跑新增/变更 Agent）

写一个测试场景：
- 初始团队 4 个 Agent 辩论"小红书AI搜索"
- 辩论中发现缺少"数据隐私"视角
- Evolver 检测到缺口，生成隐私专家 Agent
- 第二轮辩论只有隐私专家重新执行，其他 4 个复用上轮结果

给我看 evolution_log 的输出。
```

## 重要提醒

- **一次只给一个 Step 的 prompt**，等完成并验收后再给下一个
- 如果 Claude Code 中途出错或者方向跑偏，直接说"停，回到 Step X 的要求重新来"
- 每个 Step 完成后建议 `git add -A && git commit`，这样随时可以回滚
- 你的 .claude/settings.json 已经配好了权限，它不会删你的文件
- 如果它说"需要删除某个文件"，它会被 Hook 拦住，你手动在另一个终端里删就行
